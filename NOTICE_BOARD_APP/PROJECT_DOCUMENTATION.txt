                    DIGITAL NOTICE BOARD SYSTEM - PROJECT DOCUMENTATION

1. PROJECT TITLE
================================================================================
Digital Notice Board System for Educational Institutions
A comprehensive web-based application for managing placement drives, student 
registrations, and automated email notifications with real-time scheduling.

2. PROBLEM STATEMENT
================================================================================
Educational institutions face significant challenges in managing placement drives:

• Manual notification processes lead to missed opportunities for eligible students
• Lack of centralized system for tracking student responses to placement drives
• Inefficient communication between placement cell and students
• No automated reminder system for pending responses
• Difficulty in managing student eligibility criteria across multiple drives
• Time-consuming manual verification of student profiles and approvals
• Inconsistent notification delivery causing students to miss deadlines
• No systematic approach to track response rates and engagement metrics

The absence of an integrated digital solution results in:
- Students missing placement opportunities due to communication gaps
- Administrative overhead in manual tracking and follow-ups
- Inconsistent data management and reporting
- Delayed response collection affecting placement statistics

7. DATABASE DESIGN
================================================================================
Database Schema Overview:

Core Models:

User (Django Built-in):
• id (Primary Key)
• username (Unique)
• email
• password (Hashed)
• first_name, last_name
• is_active, is_staff, is_superuser
• date_joined, last_login

AdminProfile:
• id (Primary Key)
• user (OneToOne → User)
• name (CharField)
• email (EmailField, Unique)
• department (CharField)
• employee_id (CharField, Unique)
• phone (CharField)
• created_at, updated_at (DateTime)

StudentProfile:
• id (Primary Key)
• user (OneToOne → User)
• name (CharField)
• email (EmailField, Unique)
• phone (CharField)
• roll_number (CharField, Unique)
• department (CharField with Choices)
• year (CharField with Choices)
• cgpa (FloatField with Validators)
• is_approved (BooleanField)
• approved_by (ForeignKey → User)
• approved_at (DateTime)
• created_at, updated_at (DateTime)

Drive:
• id (Primary Key)
• title (CharField)
• company_name (CharField)
• description (TextField)
• min_cgpa (FloatField with Validators)
• eligible_departments (JSONField)
• eligible_year (CharField with Choices)
• last_date (DateTime)
• status (CharField with Choices: Active/Closed/Draft)
• created_by (ForeignKey → User)
• created_at, updated_at (DateTime)

DriveResponse:
• id (Primary Key)
• student (ForeignKey → User)
• drive (ForeignKey → Drive)
• response (CharField with Choices: interested/not_interested)
• responded_at (DateTime)
• Unique Constraint: (student, drive)

DriveResponseReminder:
• id (Primary Key)
• student (ForeignKey → User)
• drive (ForeignKey → Drive)
• reminder_count (IntegerField)
• last_reminder_sent (DateTime)
• created_at (DateTime)
• Unique Constraint: (student, drive)

StudentProfileChangeRequest:
• id (Primary Key)
• student (ForeignKey → User)
• requested_changes (JSONField)
• current_data (JSONField)
• status (CharField with Choices: pending/approved/rejected)
• admin_notes (TextField)
• reviewed_by (ForeignKey → User)
• reviewed_at (DateTime)
• created_at, updated_at (DateTime)

SystemSettings:
• id (Primary Key, Fixed to 1)
• email_backend (CharField)
• email_host (CharField)
• email_port (IntegerField)
• email_use_tls (BooleanField)
• email_host_user (EmailField)
• email_host_password (CharField)
• default_from_email (EmailField)
• reminder_enabled (BooleanField)
• reminder_interval_hours (IntegerField)
• reminder_interval_minutes (IntegerField)
• max_reminders (IntegerField)
• last_reminder_run (DateTime)
• site_name (CharField)
• site_url (URLField)
• updated_by (ForeignKey → User)
• updated_at, created_at (DateTime)

Relationships:
• User ↔ AdminProfile (One-to-One)
• User ↔ StudentProfile (One-to-One)
• User → Drive (One-to-Many, created_by)
• User → DriveResponse (One-to-Many, student)
• Drive → DriveResponse (One-to-Many)
• User → DriveResponseReminder (One-to-Many, student)
• Drive → DriveResponseReminder (One-to-Many)
• User → StudentProfileChangeRequest (One-to-Many, student)
• User → SystemSettings (Many-to-One, updated_by)

Indexes and Constraints:
• Unique constraints on email fields across profiles
• Composite unique constraints on response and reminder tables
• Database indexes on frequently queried fields
• Foreign key constraints with appropriate cascade rules
• Check constraints for CGPA ranges and valid choices

8. WORKFLOW
================================================================================
System Workflows:

A. Student Registration Workflow:
1. Student accesses registration page
2. Fills comprehensive profile form with academic details
3. System validates input data and checks for duplicates
4. Registration request submitted to admin for approval
5. Admin reviews student profile and supporting information
6. Admin approves/rejects with optional notes
7. System sends email notification to student about status
8. Approved students gain access to placement drive system
9. Student profile becomes active for drive notifications

B. Drive Creation and Management Workflow:
1. Admin logs into administrative dashboard
2. Creates new placement drive with company details
3. Sets eligibility criteria (CGPA, department, year)
4. Configures application deadline and drive status
5. System validates drive information and saves to database
6. Drive status set to 'Active' triggers notification process
7. System identifies eligible students based on criteria
8. Automatic notifications sent to all eligible students
9. Drive appears in student dashboards with response options

C. Student Notification and Response Workflow:
1. Student logs into system dashboard
2. System checks for eligible active drives
3. Popup notification displayed for pending responses
4. Student reviews drive details and eligibility criteria
5. Student submits response (interested/not_interested)
6. System records response with timestamp
7. Confirmation email sent to student
8. Popup notifications stop for responded drives
9. Response data available in admin analytics

D. Automated Email Reminder Workflow:
1. Background scheduler runs every minute
2. System checks last reminder run timestamp
3. Calculates time elapsed since last reminder batch
4. Compares with configured reminder interval
5. If interval exceeded, identifies pending responses
6. Filters students who haven't reached reminder limit
7. Generates personalized reminder emails
8. Sends emails via configured SMTP service
9. Updates reminder counts and last run timestamp
10. Logs email delivery status and errors

E. Profile Change Request Workflow:
1. Student initiates profile update request
2. System captures current profile data as baseline
3. Student submits requested changes via form
4. System stores changes in JSON format for review
5. Admin receives notification of pending request
6. Admin reviews current vs requested data comparison
7. Admin approves/rejects with explanatory notes
8. If approved, system updates student profile
9. Notification sent to student about decision
10. Request archived with audit trail

F. System Administration Workflow:
1. Admin accesses system settings panel
2. Configures email SMTP settings and credentials
3. Sets reminder intervals and maximum limits
4. Tests email configuration with sample email
5. Monitors email delivery logs and system health
6. Reviews student registrations and drive responses
7. Generates reports and analytics for management
8. Manages user roles and permissions
9. Performs system maintenance and updates
